# 《battle-3d-demo》方案设计文档

> **版本**：v2.0 | **日期**：2026-04-20 | **状态**：当前阶段执行基线

---

## 1. 文档概述

### 1.1 目的

本文档基于 `docs/3d对战最终需求.md`，输出当前 `battle-3d-demo` 阶段的技术方案设计，作为纯前端主战场 Demo 的实现基线。

本阶段目标不是直接建设完整联机产品，而是优先沉淀一个可运行、可展示、可扩展的 Three.js 主战场前端底座，为后续接入 `fish-island` 产品体系做准备。

### 1.2 当前阶段定位

当前阶段统一定位为：

- 纯前端主战场 Demo
- 聚焦场景、模型、动画、镜头、基础战斗体验
- 优先保证视觉还原、操作手感和资源接入稳定性
- 联机、房间、结算、积分、排行榜等能力仅做扩展预留

### 1.3 设计原则

| 原则 | 说明 |
|:-----|:-----|
| **体验优先** | 先保证主战场空间感、镜头和操作体验，再扩展复杂玩法 |
| **纯前端闭环** | 当前阶段不依赖后端即可完成核心 Demo 体验 |
| **资源稳定接入** | 模型、纹理、动画加载必须稳定，失败时必须有降级方案 |
| **参数双轨制** | 方案同时保留原版参考参数与工程落地映射参数 |
| **固定镜头规则** | 统一为锁定/解锁 + 边缘平移 + 滚轮缩放 + 空格回中 |
| **扩展友好** | 当前结构必须能平滑升级到联机、产品化与站点集成 |

### 1.4 当前阶段权威规则

- 草丛统一为 **6 组**
- Demo 阶段支持 **右键移动 + WASD 双模式**
- 正式产品方向保留 **右键移动为主**
- 镜头统一为 **固定斜 45° 俯视**
- **禁止自由拖拽**
- **禁止自由旋转**

## 2. 当前项目基础与技术栈

### 2.1 battle-3d-demo 当前项目基础

当前项目已经具备前端 Demo 的基础骨架，目录结构如下：

```
battle-3d-demo/src/
├── config/
├── game/
│   ├── BattleCanvas.tsx
│   ├── LoadingScreen.tsx
│   ├── effects/
│   ├── hooks/
│   ├── models/
│   ├── scene/
│   └── world/
├── hud/
├── store/
├── types/
└── utils/
```

当前 HUD 目录已具备：

- `TopBar.tsx`
- `SkillBar.tsx`
- `HealthManaBar.tsx`
- `Scoreboard.tsx`
- `MiniMap.tsx`
- `KillFeed.tsx`
- `HUD.tsx`

### 2.2 当前技术栈

| 类别 | 技术 | 说明 |
|:-----|:-----|:-----|
| 前端框架 | React 18 | UI 与状态驱动基础 |
| 语言 | TypeScript | 类型安全与配置表达 |
| 构建工具 | Vite 6 | 轻量开发与快速热更新 |
| 3D 引擎 | Three.js r170+ | 3D 渲染基础 |
| React Three 封装 | @react-three/fiber | React 方式组织 Three.js 场景 |
| 3D 工具库 | @react-three/drei | 常用场景与辅助能力 |
| 后处理 | @react-three/postprocessing / postprocessing | Bloom、抗锯齿等视觉增强 |
| 状态管理 | zustand | 游戏状态与渲染状态管理 |
| 样式 | TailwindCSS | HUD 与页面样式 |

### 2.3 选型结论

当前阶段仍然坚持使用 `Three.js + react-three-fiber + zustand` 的前端方案，原因如下：

- 满足真 3D 场景、骨骼动画、粒子、光照与后处理需求
- 与 React 生态集成自然，适合 HUD Overlay 方案
- 易于形成模型加载、地图配置、镜头控制、状态驱动的模块化结构
- 便于后续向 `fish-island` 前端集成迁移

## 4. 场景架构设计

### 4.1 场景层级

```
BattleCanvas
└── BattleScene
    ├── CameraController
    ├── Lights
    ├── BattleMap
    ├── Towers
    ├── Nexuses
    ├── HealthRelics
    ├── Champions
    ├── SnowParticles
    └── PostProcessing
```

### 4.2 场景组织原则

- 地图静态对象与动态对象分层
- HUD 永远独立于 3D 场景渲染管线
- 模型表现与状态数据解耦
- 所有可调参数优先沉淀到配置层，不散落在组件内部

### 4.3 地图参数双轨制

#### 原版参考参数

- 单路线对称桥面地图
- 基地位于两端
- 中央废墟形成团战核心区域
- 草丛总数统一为 6 组
- 防御塔形成蓝红推进层级

#### 工程落地映射参数

| 项目 | 工程值 |
|:-----|:-------|
| 地图尺寸 | `160 × 40 × 8` |
| 可行走桥面 | `X = -70 ~ +70`，`Z = -6 ~ +6` |
| 蓝方外塔 | `(-50, 0)` |
| 蓝方内塔 | `(-30, 0)` |
| 蓝方水晶 | `(-70, 0)` |
| 红方内塔 | `(30, 0)` |
| 红方外塔 | `(50, 0)` |
| 红方水晶 | `(70, 0)` |
| 生命圣物 | `(-10, 4)`、`(10, -4)` |

### 4.4 6 组草丛布局

| 草丛编号 | 坐标（X, Z） | 说明 |
|:---------|:-------------|:-----|
| C1 | `(-42, -4)` | 蓝方前场左草 |
| C2 | `(-42, 4)` | 蓝方前场右草 |
| C3 | `(-4, -4)` | 中央左草 |
| C4 | `(-4, 4)` | 中央右草 |
| C5 | `(42, -4)` | 红方前场左草 |
| C6 | `(42, 4)` | 红方前场右草 |

### 4.5 场景视觉要求

- 桥面：冰雪覆盖的石桥感
- 深渊：深蓝黑雾与远景虚化
- 两侧边界：围栏、岩石、冰晶残骸构成封锁感
- 粒子：雪花、寒气、微光
- 建筑：冰晶与霜石结构，阵营颜色可辨识

## 5. 镜头与输入系统设计

### 5.1 镜头系统目标

镜头必须服务于 ARAM 主战场体验，而不是自由探索体验。

### 5.2 镜头规则

- 固定斜 45° 俯视
- 支持锁定/解锁
- 支持边缘平移
- 支持滚轮缩放
- 支持空格回中
- 不允许自由拖拽
- 不允许自由旋转

### 5.3 镜头实现建议

#### 锁定模式

- 相机目标绑定当前主控英雄
- 英雄位于画面中央略偏下位置
- 仅允许缩放，不允许边缘平移推动主镜头目标脱离英雄

#### 解锁模式

- 相机目标从英雄位置解耦
- 鼠标靠近屏幕边缘触发平移
- `Space` 立即回中到主控英雄

### 5.4 输入系统方案

#### 当前 Demo 阶段

- 右键点击地面移动
- WASD 键盘辅助移动
- `Q/W/E/R` 技能热键
- `D/F` 召唤师技能预留位
- `S` 停止动作
- `Y` 切换锁定/解锁
- `Space` 回中

#### 输入冲突处理原则

- 鼠标右键移动与 WASD 不可出现状态抢占死锁
- 相机输入优先级低于核心角色输入
- 技能释放时需要明确取消逻辑与目标指示反馈

## 6. 角色、模型与动画系统设计

### 6.1 目标

在只提供模型文件、纹理和动画资源的前提下，稳定完成英雄接入，确保不会因单个资源异常而破坏整个场景。

### 6.2 当前模型接入原则

- 统一模型目录规范
- 模型加载必须可缓存
- 模型必须进行尺寸归一化
- 骨骼模型必须安全克隆
- 动画控制器必须独立实例化

### 6.3 推荐资源规格

| 类型 | 建议 |
|:-----|:-----|
| 模型格式 | `glTF / GLB` |
| 动画数量 | `idle / run / attack / cast / death` |
| 纹理大小 | `512 × 512` 或 `1K` 轻量贴图 |
| 面数目标 | 单英雄 `2000 - 5000` 面优先 |

### 6.4 降级策略

- 模型缺失：降级为程序化英雄模型
- 动画缺失：至少保留 idle / run 基础表现
- 纹理缺失：降级为基础材质
- 单英雄加载失败：不得中断整个场景渲染

### 6.5 动画状态机

```
idle <-> run
run -> attack -> idle
run -> cast -> idle
any -> death -> revive -> idle
```

### 6.6 当前阶段表现要求

- 英雄必须有阵营识别
- 英雄必须支持头顶血条/名牌/选中光环
- 动画切换必须平滑
- 同屏多个英雄必须保持较稳定性能

## 7. 基础战斗与交互系统设计

### 7.1 当前阶段最小战斗闭环

当前阶段至少需要跑通以下闭环：

```
移动 -> 接敌 -> 普攻 -> 受击反馈 -> 死亡/复活 -> 回到战场
```

### 7.2 普攻系统

- 支持靠近目标后自动或触发式普攻
- 需要具备前摇、命中、受击反馈
- 近战与远程表现可先共用数据结构，逐步细化

### 7.3 技能系统

当前阶段不要求完成完整英雄池技能，但必须建立统一技能骨架：

- 技能槽位：`Q/W/E/R`
- 技能数据结构：名称、冷却、施法类型、目标类型、范围、特效占位
- 技能交互：按键触发、指示器预留、取消逻辑、冷却显示

### 7.4 防御塔系统

- 具备攻击范围识别
- 具备目标选取与攻击表现
- 具备受击、掉血、破损反馈预留位
- 当前阶段先实现基础交互逻辑，不追求正式服全部优先级细节

### 7.5 草丛系统

- 草丛可穿行
- 草丛提供视野遮蔽
- 草丛内攻击/施法可短暂暴露位置
- 视觉表现不应仅为调试色块，最终需转向灌木/冰雪植被形态

### 7.6 生命圣物系统

- 靠近触发拾取
- 提供生命/法力恢复
- 存在重生计时
- 视觉表现为悬浮发光物或符文装置

### 7.7 死亡与复活

- 英雄死亡切换至 death 表现
- HUD 可显示复活倒计时
- 复活后回到基地出生点
- 复活行为与镜头回中逻辑需保持一致

## 8. HUD 与前端状态设计

### 8.1 HUD 分层原则

- 所有复杂界面全部走 React Overlay
- Three.js 场景仅负责 3D 视觉对象
- HUD 不应因为 3D 帧更新而被迫整体重渲染

### 8.2 当前阶段 HUD 组成

| 模块 | 目标 |
|:-----|:-----|
| `TopBar` | 顶部状态与节奏信息 |
| `SkillBar` | 技能栏与冷却反馈 |
| `HealthManaBar` | 自身血蓝信息 |
| `Scoreboard` | 最小计分信息 |
| `MiniMap` | 战场整体方位感 |
| `KillFeed` | 关键事件提示 |
| `LoadingScreen` | 场景与资源加载反馈 |

### 8.3 状态管理建议

当前阶段统一采用 `zustand` 管理以下状态：

- 场景加载状态
- 英雄状态
- 建筑状态
- 圣物状态
- HUD 交互状态
- 镜头模式
- 输入模式与技能状态

### 8.4 状态与渲染解耦原则

- Store 负责可序列化状态
- Three.js 对象负责表现缓存与瞬态动画
- 高刷更新逻辑尽量在帧循环与本地对象层完成
- 避免将所有三维细节都塞回 React 状态导致重渲染成本过高

## 11. 性能、兼容性与风险控制

### 11.1 前端性能策略

| 策略 | 说明 |
|:-----|:-----|
| 几何体合并 | 静态桥面、围栏、装饰尽量合批 |
| 实例化渲染 | 雪花、重复装饰物优先实例化 |
| 模型归一化缓存 | 预处理后复用，避免重复计算 |
| 动画更新裁剪 | 只更新必要对象动画 |
| 纹理压缩 | 支持轻量纹理与后续 KTX2 路线 |
| 后处理降级 | 低配模式关闭或弱化 Bloom/阴影 |
| 粒子预算 | 控制粒子数量与材质复杂度 |

### 11.2 兼容性目标

- 桌面端 Chrome / Edge 优先
- 集显办公机可运行
- 默认不以移动端为主目标

### 11.3 当前阶段主要风险

| 风险 | 概率 | 应对 |
|:-----|:-----|:-----|
| 模型资源规格不统一 | 高 | 统一归一化与降级策略 |
| 动画命名不统一 | 高 | 做动画映射表与回退机制 |
| 场景性能不足 | 中 | 提前建立低配降级开关 |
| 镜头与输入冲突 | 中 | 提前统一输入状态机 |
| 草丛与视野表现不自然 | 中 | 先逻辑正确，再逐步优化视觉表现 |
| 战斗范围膨胀 | 高 | 坚持当前阶段仅做基础闭环 |

1. 迁移前端主战场页面
2. 对接站点路由与壳层
3. 对接登录态与用户身份
4. 对接 WebSocket / 对局服务
5. 对接积分、战绩和个人面板

---

## 13. 结论

当前阶段最正确的路线，是先围绕 `battle-3d-demo` 打磨一个高质量的纯前端主战场 Demo，把地图、镜头、输入、模型动画和基础战斗体验做扎实。

该方案的核心价值在于：

- 先验证最重要、最核心的对战视觉与操作体验
- 先建立稳定的模型接入与渲染基座
- 为后续联机化、产品化、站点集成预留清晰的升级路径

因此，本方案文档自本版本起，作为当前 `battle-3d-demo` 阶段的技术执行基线。
