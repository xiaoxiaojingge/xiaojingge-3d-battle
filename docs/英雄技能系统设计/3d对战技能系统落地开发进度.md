# 3d对战技能系统落地开发进度

## 1. 文档说明

本文档用于记录本轮“3d 对战技能系统正式落地”的实际开发进度，目标是保证：

- 每个阶段的实现边界清晰可追踪
- 新开会话后可以快速恢复上下文

## 2. 本轮确认范围

### 2.1 总体目标

本轮按《3D对战技能系统方案设计文档》进行较大范围正式落地，尽量覆盖文档主体能力，并以亚索作为唯一强制验收样板。

### 2.2 已确认实施边界

1. 按文档做尽量完整的大范围落地，而不是临时 Demo 级补丁。
2. 英雄范围为“亚索完整实现 + 其他英雄通用基础模板接入”。
3. 本轮新建独立 `battle-3d-demo-server`，技术栈与 `fish-island-backend` 保持一致方向。
4. 本轮以亚索为唯一强制验收样板。
5. 接受抽出独立共享层，作为前后端共享协议与配置载体。

## 6. 关键设计决策记录

### 决策 1：共享层独立存放

共享层单独放在仓库根目录下的 `battle-3d-demo-shared`，用于承载：

- 协议版本
- 技能定义
- 状态定义
- 共享常量
- 英雄技能配置

### 决策 2：共享层优先使用 JSON / 文档化结构

考虑到前端为 TypeScript、服务端为 Java，真正长期稳定共享的内容优先采用配置文件与协议文档，而不是强行共用同一份源码实现。

### 决策 3：服务端独立工程化

`battle-3d-demo-server` 作为独立正式服务端，后续可以平滑吸收进 `fish-island` 体系，不再继续依赖当前前端目录内的 Node + Socket.IO 演示服务作为正式技能系统承载。

### 决策 4：亚索作为强制上限验证样板

亚索覆盖状态驱动、形态切换、投射物、拦截、位移、终极技能目标限制等复杂场景，适合作为系统强制验收样板。

## 7. 风险与待关注事项

- 当前任务范围非常大，必须优先保证“共享协议 -> 服务端骨架 -> 客户端接入”的顺序，不可反过来做。
- 后续如果修改 `src/types/game.ts` 或 `src/config/gameConfig.ts`，必须补全中文注释。
- 现有 `battle-3d-demo/server/socketServer.js` 是演示原型，后续需要逐步降级为旧方案或调试方案，避免与正式服务端职责混淆。
- 客户端现有 `SkillBar` 与调试热键链路不是正式技能系统，后续改造时要避免继续沿用“动画驱动技能”的错误方向。

  - **spellCastAccepted**：根据 requestId 调用 `acceptLocalSpellPrediction()`，绑定服务端 castInstanceId。
  - **spellCastRejected**：根据 requestId 查找预测记录，精确回滚动画（恢复 idle）、清理预测表、如有瞄准快照则恢复瞄准态供玩家重试。
  - **spellCastStarted**：调用 `markLocalSpellPredictionStarted()`，对本机已预测过的施法跳过重复动画和语音播放（shouldSkipLocalReplay）。
  - **spellStageChanged (interrupted)**：清理预测记录，如有瞄准快照恢复瞄准态，否则退出瞄准。
  - **spellStageChanged (finished)**：正常清理预测记录。

- **设计要点**：
  - 预测表以 `requestId` 为主键，`castInstanceId` 通过 accepted 事件写入后作为服务端事件的关联键。
  - 瞄准快照在登记时拍摄，仅在拒绝/中断且为受控英雄时恢复，不会对其他玩家产生副作用。
  - 本机预测施法的 started 事件跳过动画/语音重放，避免客户端预测造成双重播放。

- **逻辑自检**：

  - 确认 accepted → started 流程中 castInstanceId 正确传递。
  - 确认 finished / interrupted 正确清理预测记录，无泄漏。
  - 确认 shouldSkipLocalReplay 仅对受控英雄生效，不影响对手施法的表现。

- **性能自检**：
  - 预测表为简单的 `Record<string, ...>`，通常只有 0~2 条记录，查找和写入无性能瓶颈。
  - 所有 action 采用浅拷贝 + delete，无全量遍历。
