# 3D对战技能系统方案设计文档

> **文档定位**：正式产品长期架构设计文档  
> **适用范围**：`battle-3d-demo` 前端主战场底座、未来 `fish-island` 正式项目对战模块  
> **设计目标**：构建一套适用于所有英雄、支持长期扩展、可上线落地、尽可能参考《英雄联盟》真实战斗机制的 3D 对战技能系统  
> **当前状态**：第一版正式方案文档

---

## 1. 文档概述

### 1.1 背景

当前仓库中已经存在一个可运行的 `battle-3d-demo`，它已经完成了 3D 主战场展示、英雄模型加载、动画切换、镜头控制、HUD 基础布局、局部联机同步等前端底座能力。这套底座对于“技能表现层”已经具备良好的承载条件，但它尚不具备“正式产品级技能系统”的核心能力。

从正式产品视角看，技能系统不能只是“按下 Q 播放一段动画”，而必须是一套完整的战斗规则系统。它必须统一覆盖以下内容：

- 被动技能
- 普攻与普攻强化
- 英雄主动技能 `Q/W/E/R`
- 召唤师技能 `D/F`
- 装备主动
- Buff / Debuff
- 护盾 / 治疗 / 持续伤害
- 位移 / 击退 / 击飞 / 压制 / 禁锢 / 沉默 / 缴械
- 投射物 / 轨迹技能 / 范围技能 / 持续施法 / 引导技能 / 可打断技能
- 地形阻挡、风墙类拦截、碰撞体、技能命中与伤害裁决
- 客户端预测、服务端权威同步、回滚纠正与反作弊校验

因此，这份文档不是为某个英雄单独设计技能逻辑，而是要设计一套**全英雄通用、长期可维护、可配置、可扩展、可上线**的技能系统总方案。

### 1.2 文档目标

本文档的目标有四个。

第一，定义一套正式产品级的 3D 对战技能系统总体架构，明确系统边界、模块划分、职责归属与演进方向。

第二，建立统一的技能领域抽象，使新英雄接入时只需要新增配置、效果逻辑和少量专属处理器，而不需要频繁修改核心战斗框架。

第三，明确当前 `battle-3d-demo` 与未来正式项目之间的关系，说明哪些能力可以直接复用，哪些能力必须在正式项目中补齐。

第四，以亚索为完整示例，说明一个高复杂度英雄应如何接入这套系统，并保证模型演绎、技能判定、伤害结算和联机同步都正确工作。

### 1.3 核心设计原则

#### 1.3.1 服务端权威

正式产品中，所有战斗逻辑必须以服务端为权威源。客户端可以做输入预判、动画预播、轨迹预显和局部预测，但冷却、资源、命中、伤害、状态、位移、控制、死亡、击杀归属与战斗日志必须由服务端裁定。

#### 1.3.2 表现与规则解耦

技能系统必须拆分为“规则层”和“表现层”。规则层负责决定技能能否释放、何时生效、命中谁、造成什么效果；表现层负责动画、特效、音效、镜头、UI、震屏、受击反馈等视觉与听觉呈现。两者通过统一协议和统一事件模型衔接，而不是互相写死。

#### 1.3.3 通用抽象优先于英雄特化

系统设计必须先支持普遍技能范式，再处理特殊英雄个例。复杂英雄不能通过在核心引擎中堆叠 `if (heroId === 'yasuo')` 方式实现，而应通过：

- 统一元数据配置
- 通用效果原子
- 可扩展目标选择器
- 技能专属逻辑处理器
- 可插拔战斗事件监听

来表达差异化行为。

#### 1.3.4 配置驱动 + 原子效果 + 专用处理器混合模式

纯 JSON 配置无法完整表达所有 MOBA 技能交互，纯代码写死又会导致扩展困难。因此正式方案采用混合模式：

- 通用参数放在配置层
- 可组合效果放在效果原子层
- 极特殊机制放在技能专属处理器层

这套模式既能支持大多数技能快速接入，也能承载亚索风墙、击飞联动、二段机制这类复杂英雄逻辑。

#### 1.3.5 参考《英雄联盟》原作机制，但采用工程映射实现

系统目标是尽可能参考《英雄联盟》原始技能体验与规则思想，包括技能时序、前后摇、命中类型、控制系统、地形阻挡、技能标签、技能联动、状态覆盖、客户端表现反馈等；但工程实现必须面向当前产品技术栈与网络模型进行映射，而不是机械复制原作内部实现。

## 2. 当前现状评估

### 2.1 `battle-3d-demo` 当前已具备的能力

从当前代码结构看，`battle-3d-demo` 已经具备作为技能系统前端表现底座的必要基础。

#### 2.1.1 动作槽位与基础动作映射已存在

当前项目已经存在统一的动作槽位抽象：

- `basicAttack`
- `q`
- `w`
- `e`
- `r`
- `recall`

并且 `gameConfig.ts` 与 `heroConfig.ts` 已经可以根据不同英雄把动作槽位映射到模型实际动画片段，例如亚索 `q -> Spell1A`、`w -> Spell2_0`、`e -> Spell3`、`r -> Spell4`。这意味着当前前端已经具备“技能表现挂点”的雏形。

#### 2.1.2 动画请求链路已存在

当前系统已有 `AnimationClipRequest`、`AnimationController`、`Champion.tsx` 等结构，可以根据请求播放指定片段、设置播放倍率、持续时间、回退状态与移动锁定。它虽然还不是正式技能系统的一部分，但足以复用为“技能表现执行层”的一环。

#### 2.1.3 语音与表现反馈链路已存在

当前项目已经存在按动作槽位触发语音的基础能力，并支持远端同步语音请求。这为正式技能系统中的施法语音、命中语音、强化语音、表情轮盘语音提供了已有基础。

#### 2.1.4 联机表现同步底座已存在

当前 demo 通过 Socket.IO 搭起了一个轻量联机同步方案，可以同步：

- 角色移动
- 动画请求
- 表情请求
- 语音请求
- 角色快照

这套方案本身还不够承载正式产品，但它证明了当前前端已经具备“接收权威战斗事件并进行表现”的基础结构。

#### 2.1.5 HUD 和输入入口已预留

当前 HUD 中已有 `SkillBar`，虽然仍然是静态展示数据，但 UI 位置与交互形态已经存在。`InputController` 也已经具备输入采集能力，可以继续扩展技能输入、施法模式切换、范围指示与目标确认。

### 2.2 当前缺失的关键能力

当前 demo 与正式产品技能系统之间，真正的差距并不在“动画资源”，而在“战斗规则本体”。

#### 2.2.1 缺失统一技能定义系统

当前没有正式的 `SkillDefinition`、`SpellDefinition`、`PassiveDefinition`、`SummonerSpellDefinition`、`ActiveItemDefinition` 等统一定义模型。现有 `Q/W/E/R` 仅仅是动画槽位，不是技能实体。

#### 2.2.2 缺失技能生命周期管理

当前没有正式的施法流程模型，无法表达：

- 按键进入瞄准
- 请求施法
- 服务端验证
- 前摇
- 弹体生成
- 持续体存在
- 命中事件
- 收尾
- 中断
- 二段释放

#### 2.2.3 缺失 Buff / Debuff / 控制系统

当前没有正式的状态系统，因此无法统一表达沉默、击飞、减速、加速、禁锢、护盾、持续伤害、持续治疗、强化普攻、不可选中、风墙拦截等机制。

#### 2.2.4 缺失命中判定与伤害裁决体系

当前系统没有正式的碰撞体、命中盒、范围体、投射物、穿透规则、护盾吸收、伤害公式、技能标签修正等机制，因此无法形成真实战斗规则。

#### 2.2.5 缺失服务端战斗权威引擎

当前 `battle-3d-demo` 服务端更多是演示用途的同步服务器，还不是正式产品的战斗引擎。未来正式项目必须在 `fish-island` 产品体系内补齐专门的战斗房间、战斗帧推进、技能执行器、状态系统、同步协议和裁决链路。

### 2.3 与正式产品的关系判断

结合仓库现状，可以得出清晰结论：

`battle-3d-demo` 的定位应当是正式技能系统的**前端主战场表现底座**，而不是正式技能系统本体。它可以复用为：

- 输入采集层
- 技能范围预览层
- 动画表现层
- 特效挂载层
- 语音与镜头反馈层
- HUD 展示层
- 客户端预测表现层

而正式产品必须新增：

- 服务端战斗模拟层
- 技能定义与加载层
- 技能执行与效果系统
- 命中与伤害计算系统
- Buff / Debuff / 控制系统
- 投射物与空间判定系统
- 权威同步与回滚纠正机制

## 3. 设计目标与非目标

### 3.1 设计目标

#### 3.1.1 支持全英雄扩展

技能系统必须支持后续任意英雄接入，而不是只适配亚索。新增英雄的理想接入方式应为：

- 新增英雄基础战斗配置
- 新增技能定义配置
- 新增技能表现配置
- 如有必要，补充少量专属逻辑处理器

而不需要修改核心战斗流程、统一判定框架或协议结构。

#### 3.1.2 支持完整技能类型覆盖

系统必须覆盖正式对战中常见的全部技能范式，包括：

- 被动触发类
- 普攻链与普攻重置类
- 指向单位技能
- 指向地点技能
- 方向型技能
- 无目标技能
- 引导技能
- 持续施法技能
- 二段释放技能
- 切换技能
- 蓄力技能
- 投射物技能
- 地形技能
- 召唤技能

#### 3.1.3 支持正式联机与反作弊

技能系统必须天然适配正式联机环境，所有关键判定都必须支持服务端权威与防作弊验证。不能让客户端提交“我已经命中了谁”“我造成了多少伤害”这类结果型指令，客户端只能提交输入意图与必要瞄准数据。

#### 3.1.4 支持复杂技能联动

系统必须支持：

- 技能与被动联动
- 技能与普攻联动
- 技能之间的状态依赖
- Buff 对技能形态的修改
- 装备主动对技能链的插入
- 风墙、护盾、击飞、压制、不可阻挡等特殊交互

#### 3.1.5 支持长期数值迭代

数值、冷却、成长、系数、状态时长、目标过滤规则、资源消耗、技能标签等应尽量配置化，以便未来进行平衡性调整、热更新控制和版本管理。

### 3.2 非目标

#### 3.2.1 非目标一：首版不追求完整通用技能编辑器

本文档会定义配置结构与系统抽象，但不要求首版必须同时交付一个复杂的可视化技能编辑器。编辑器属于后续效率工程，不是第一优先级。

#### 3.2.2 非目标二：不把前端做成战斗权威端

前端可以预测与预播，但不会成为最终权威裁决端。凡是影响战斗胜负、状态合法性和反作弊的能力，都必须落在服务端。

#### 3.2.3 非目标三：不以单英雄特化堆砌系统

亚索会作为完整示例，但系统设计不会以“为亚索硬写一套专门逻辑”作为核心方案。亚索只是用来验证系统上限，而不是反过来绑架系统结构。

## 4. 总体架构设计

### 4.1 总体分层

正式产品建议采用六层结构。

#### 4.1.1 客户端输入与表现层

这一层部署在 `battle-3d-demo` 或未来正式前端中，负责：

- 输入采集
- 技能选中与施法模式切换
- 技能范围预览
- 动画播放
- 特效与音效表现
- HUD 展示
- 客户端预测
- 回正后的表现修复

#### 4.1.2 共享协议与配置层

这一层由前后端共享，负责统一描述：

- 技能元数据
- 状态定义
- 战斗事件结构
- 网络协议结构
- 枚举、标签、目标过滤规则
- 版本兼容字段

共享层的目标是保证前后端对同一技能实体、同一状态类型、同一事件结构有一致理解。

#### 4.1.3 服务端战斗会话层

这一层运行在正式项目的实时战斗服务中，负责：

- 战斗房间生命周期
- 玩家接入与断线处理
- 战斗 Tick 推进
- 战斗内实体注册
- 状态同步组织
- 战斗日志与回放基础记录

#### 4.1.4 服务端技能执行层

这一层是技能系统本体，负责：

- 施法合法性校验
- 生成技能实例
- 驱动技能生命周期
- 执行效果链
- 管理投射物与持续体
- 触发命中与状态事件
- 处理打断与二段逻辑

#### 4.1.5 服务端规则计算层

这一层负责：

- 伤害与治疗计算
- Buff / Debuff 计算
- 状态叠层与覆盖
- 护盾吸收
- 控制抗性修正
- 移动与位移合法性裁决
- 地形阻挡与特殊碰撞逻辑

#### 4.1.6 服务端同步广播层

这一层负责把权威结果广播给客户端，包括：

- 技能开始
- 技能中断
- 投射物生成
- 命中结果
- Buff 变化
- 伤害与治疗结果
- 位移与击飞结果
- 冷却与资源变化
- 最终状态快照与增量事件

### 4.2 推荐的正式集成关系

在正式产品中，推荐采用如下关系：

- `battle-3d-demo` 演进为正式前端主战场模块
- `fish-island` 保留主业务站点、账号、房间、社交、外围业务能力
- 新增专门的实时战斗服务或在现有后端体系中拆分战斗模块
- 通过 Netty / WebSocket 承载高频战斗事件同步
- 通过共享协议层保证前后端一致性

### 4.3 架构核心结论

整体上，这套技能系统不是“前端技能播放系统”，也不是“单个英雄脚本集合”，而是一套围绕**权威战斗事件流**建立的正式产品架构。客户端关心“如何表现”，服务端关心“是否合法、是否命中、造成什么结果”，共享层关心“双方是否对同一规则使用同一语言”。

这也是整份文档后续所有详细设计的出发点。

## 5. 核心统一抽象总览

技能系统要真正做到全英雄通用，必须先统一战斗域中的核心对象，再往上挂技能逻辑与表现逻辑。正式方案中，建议所有复杂度最终收敛到以下几类核心对象：

- 战斗实体
- 技能定义
- 技能实例
- 状态效果
- 投射物与区域体
- 命中事件
- 结果事件

后续章节会逐一展开这些抽象，并说明它们如何承载普攻、被动、召唤师技能、装备主动以及亚索这类高复杂英雄机制。

### 5.1 战斗实体抽象

技能系统中的一切规则都必须围绕“战斗实体”展开。正式方案中，不应只把英雄视为唯一技能目标，而应统一抽象成可参与战斗交互的实体对象。

建议战斗实体至少包括以下几类：

- 英雄实体
- 小兵或 AI 单位实体
- 召唤物实体
- 投射物实体
- 可交互地形实体
- 临时区域实体
- 建筑或特殊机关实体

每个战斗实体都应具备统一的基础字段，例如：

- 唯一实例 ID
- 实体类型
- 所属阵营
- 空间位置与朝向
- 当前生命、护盾、资源
- 基础属性与实时属性快照
- 可选中状态、可碰撞状态、可见性状态
- 当前 Buff / Debuff 列表
- 当前施法状态
- 当前移动状态
- 当前死亡状态

统一实体抽象的价值在于：技能系统不需要关心“这是不是英雄”，只需要关心“这是不是一个满足目标过滤条件的战斗实体”。

### 5.2 技能定义抽象

技能不是一个按钮，也不是一段动画，而是一份完整的规则定义。正式系统建议把技能定义拆成五层信息。

#### 5.2.1 技能基础元数据

用于标识技能身份和外部引用关系。

建议包含：

- `skillId`
- `heroId`
- `slot`
- `name`
- `tags`
- `maxLevel`
- `icon`
- `description`
- `isPassive`

其中 `slot` 不只包含 `Q/W/E/R`，还必须统一支持：

- `passive`
- `basicAttack`
- `q`
- `w`
- `e`
- `r`
- `summonerD`
- `summonerF`
- `itemActive1...n`

#### 5.2.2 施法元数据

用于描述技能如何被释放。

建议包含：

- 目标类型
- 施法类型
- 前摇时长
- 后摇时长
- 是否需要选点
- 是否需要选目标
- 是否允许移动中释放
- 是否会锁移
- 是否会打断普攻
- 是否允许被打断
- 是否为二段技能
- 是否为引导技能
- 是否为持续施法技能
- 是否为蓄力技能

#### 5.2.3 数值元数据

用于描述技能成长与消耗。

建议包含：

- 基础冷却与等级成长
- 基础消耗与等级成长
- 基础射程
- 基础宽度 / 半径 / 扇形角度
- 基础速度 / 飞行时间
- 基础持续时间
- 基础伤害 / 治疗 / 护盾值
- 属性加成系数
- 可充能次数与回复规则

#### 5.2.4 规则元数据

用于描述技能如何参与规则判定。

建议包含：

- 命中次数规则
- 是否穿透
- 是否首目标停止
- 是否会被地形阻挡
- 是否会被风墙类技能拦截
- 目标过滤器
- 阵营过滤器
- 视野要求
- 是否允许对无敌单位释放
- 是否只能对击飞目标释放
- 是否允许穿墙

#### 5.2.5 表现元数据

用于客户端表现层使用。

建议包含：

- 动画槽位
- 动画片段名
- 特效资源 ID
- 音效资源 ID
- 命中特效资源 ID
- HUD 指示器样式
- 施法范围预览样式
- 镜头反馈策略
- 震屏与受击反馈策略

### 5.3 技能类型统一分类

正式系统中不建议只靠一个 `type` 枚举硬分所有技能，而是用多个维度共同定义技能范式。

#### 5.3.1 按目标方式分类

- 无目标技能
- 指向单位技能
- 指向地点技能
- 指向方向技能
- 自身中心技能
- 锁定目标后位移技能

#### 5.3.2 按施法过程分类

- 瞬发技能
- 前摇施法技能
- 持续施法技能
- 引导技能
- 蓄力技能
- 切换技能
- 二段释放技能

#### 5.3.3 按命中载体分类

- 直接作用型
- 范围生效型
- 线性弹体型
- 抛物弹体型
- 锁定跟踪型
- 持续区域型
- 召唤物触发型
- 地形实体型

#### 5.3.4 按功能定位分类

- 伤害技能
- 治疗技能
- 护盾技能
- 位移技能
- 控制技能
- 强化技能
- 召唤技能
- 反制技能
- 视野技能

一个技能可以同时拥有多个标签。例如亚索 `Q3` 既可以是“方向型技能”，也是“线性弹体型技能”，同时还是“伤害 + 击飞型技能”。

### 5.4 技能实例抽象

技能定义是静态模板，而技能实例是战斗中的一次具体释放。正式系统中必须把二者严格区分。

每次成功发起施法后，都应创建一个 `SkillCastInstance`。它至少应记录：

- 施法实例 ID
- 技能定义 ID
- 施法者实体 ID
- 当前阶段
- 施法开始时间
- 输入参数快照
- 当次施法锁定的目标或方向
- 已经命中的实体集合
- 当前生成的投射物集合
- 当前生成的持续区域集合
- 当前是否已被中断
- 当前是否已进入结算结束阶段

技能实例存在的意义在于让系统可以正确处理：

- 多段技能的分阶段触发
- 引导过程中的周期 Tick
- 二段释放的重入窗口
- 范围持续体的多次命中
- 中断时的残留效果清理

### 5.5 Buff / Debuff / 状态效果抽象

正式技能系统中，状态效果不应只是“一个数组里的字符串”，而必须是可计算、可叠层、可覆盖、可过期、可驱动事件的正式对象。

建议统一抽象为 `StatusEffectInstance`，并包含：

- 状态实例 ID
- 状态定义 ID
- 来源实体 ID
- 挂载目标实体 ID
- 状态类型
- 状态层数
- 状态强度
- 生效开始时间
- 持续时间
- 覆盖规则
- 刷新规则
- 移除条件
- 与其他状态的互斥规则

### 5.6 普攻、被动、召唤师技能、装备主动的统一接入方式

正式系统必须避免把普攻、被动、召唤师技能、装备主动放在技能系统之外单独维护。正确做法是让它们共享同一套抽象，再通过槽位、类型与触发方式做区分。

#### 5.6.1 普攻

普攻应被视为一种特殊的 `basicAttack` 技能定义。它同样具备：

- 施法前摇
- 抬手点
- 弹道或命中点
- 攻击范围
- 暴击标签
- 普攻附带效果
- 被动与装备联动

#### 5.6.2 被动

被动不通过玩家主动施法触发，但仍然应该拥有正式定义。被动通常由战斗事件驱动，例如：

- 命中时触发
- 暴击时触发
- 位移后触发
- 释放技能后触发
- 达成层数后触发
- 进入特定状态时触发

#### 5.6.3 召唤师技能 `D/F`

召唤师技能本质上仍然是可释放技能，只是其来源不是英雄自身，而是玩家对局携带配置。它应和英雄技能共享同一套施法、命中、伤害、冷却与同步框架。

#### 5.6.4 装备主动

装备主动建议被视为从装备系统注入的外部技能定义。它们的输入、冷却、施法条件、命中与结算流程应完全复用技能系统，而不是另做一套“物品主动系统”。

## 6. 技能生命周期与战斗时序

### 6.1 生命周期分阶段模型

一套正式可扩展的技能系统，必须先定义清晰统一的生命周期阶段。推荐把一次技能释放拆成如下阶段：

1. 输入采集阶段
2. 本地预检查阶段
3. 请求发送阶段
4. 服务端合法性校验阶段
5. 前摇阶段
6. 生效触发阶段
7. 持续执行阶段
8. 收尾阶段
9. 冷却运行阶段

这九个阶段并不要求每个技能都完整经历，但系统必须有能力表达它们。

### 6.2 输入采集阶段

客户端负责采集技能输入，并根据技能类型决定是否进入：

- 直接释放
- 目标选择
- 地点选择
- 方向瞄准
- 二段确认

客户端此时只做“表现级合法性预检查”，例如：

- 当前技能是否在冷却中
- 当前是否处于明显无法施法状态
- 当前鼠标指向是否超出最大施法范围

但这一步不是权威裁决，只是为了优化交互反馈。

### 6.3 服务端合法性校验阶段

服务端收到施法请求后，需要进行完整合法性验证。至少包括：

- 施法者是否存在
- 当前是否存活
- 当前是否被沉默、压制或其他禁止施法状态限制
- 当前技能是否在冷却
- 资源是否足够
- 目标是否存在且合法
- 目标是否在有效范围内
- 是否满足技能特殊前置条件
- 是否满足当前技能形态前置条件

如果验证失败，服务端返回施法失败事件，客户端取消预测表现或播放失败反馈。

### 6.4 前摇与施法锁定阶段

验证成功后，技能进入前摇阶段。这个阶段的本质是：技能已被接受，但尚未正式生效。此时系统需要决定：

- 是否锁移
- 是否锁转身
- 是否允许普攻重置
- 是否会打断当前引导
- 是否允许被控制打断
- 动画何时开始播放
- 技能关键帧何时到达

正式系统建议把“技能关键生效时间点”与“动画开始时间点”分开管理，避免所有技能都以动画开始即视为命中。

### 6.5 生效触发阶段

当技能到达关键生效时间点时，系统进入正式生效阶段。根据技能类型不同，可能发生以下行为：

- 立即结算目标伤害
- 生成投射物
- 生成持续区域
- 施加 Buff / Debuff
- 启动引导
- 执行位移
- 创建地形实体

### 6.6 持续执行阶段

对于引导技能、持续技能、投射物技能、持续区域技能，技能在生效后通常不会立刻结束，而是进入持续执行阶段。这个阶段中系统需要持续推进：

- 周期 Tick 伤害
- 投射物位置更新
- 区域体过期判断
- 多次命中判定
- 引导是否中断
- 二段技能是否开放释放窗口

### 6.7 中断机制

正式技能系统必须把中断视为一级机制，而不是附加处理。中断来源通常包括：

- 死亡
- 眩晕 / 压制 / 击飞
- 位移被强制打断
- 主动取消
- 技能形态切换
- 距离或视野条件失效

被中断后，系统必须明确：

- 当前技能实例是否立即终止
- 已生成投射物是否继续存在
- 已生成区域体是否继续存在
- 冷却是否进入全额、部分或不进入
- 消耗是否返还或不返还

### 6.8 二段技能与重入窗口

二段技能应通过“同一技能定义 + 不同施法阶段状态”实现，而不是硬拆成两个完全无关技能。系统应支持：

- 第一次释放后开启二段窗口
- 二段窗口超时回收
- 二段释放条件检查
- 二段释放时复用同一技能链路

这对于亚索 `Q` 层数成长、某些英雄蓄力/引爆类技能、以及召唤师技能衍生形态都很重要。

## 7. 技能效果系统设计

### 7.1 设计原则

技能系统中的大部分结果，不应直接写在技能本身里，而应通过“效果原子”执行。技能负责组织时序和上下文，效果系统负责实际结算。

### 7.2 建议的效果原子类型

正式系统建议至少支持以下效果原子：

- `Damage`
- `Heal`
- `Shield`
- `ApplyBuff`
- `RemoveBuff`
- `ModifyResource`
- `Displace`
- `Knockback`
- `Knockup`
- `Pull`
- `Dash`
- `Teleport`
- `SpawnProjectile`
- `SpawnArea`
- `SpawnTerrain`
- `RevealVision`
- `PlayTrigger`

### 7.3 效果执行上下文

每个效果原子在执行时，都必须拿到统一上下文，而不是任意读写全局状态。建议上下文至少包含：

- 施法者
- 当前技能实例
- 当前命中目标
- 当前命中点位置
- 当前技能等级
- 当前快照属性
- 当前标签集合
- 当前已命中历史
- 当前战斗时间

这样可以保证同一个效果原子既可用于英雄技能，也可用于召唤师技能与装备主动。

### 7.4 组合效果链

一个技能通常不是单一效果，而是一条效果链。例如一个技能可以同时：

- 造成伤害
- 施加减速
- 生成持续区域
- 触发被动计层

因此，技能定义中应允许按阶段挂载效果链，例如：

- 施法时效果
- 命中时效果
- 周期 Tick 效果
- 结束时效果
- 被中断时效果

### 7.5 事件监听与被动触发

除了显式效果链，正式系统还应支持“战斗事件监听器”模式，以承接被动与外部修饰。例如：

- 命中后叠层
- 受击后刷新护盾
- 施法后强化下次普攻
- 终极技能命中后改变护甲穿透状态

这类能力不应全部塞进单个技能类里，而应由统一事件系统驱动。

### 7.6 为什么仍然需要专用处理器

虽然效果原子可以覆盖大部分技能，但仍有少量复杂技能需要专用处理器，例如：

- 风墙拦截飞行物
- 只能对击飞目标释放的终结技能
- 与历史命中目标强耦合的位移技能
- 跟随上一技能状态改变形态的技能

因此，正式系统建议支持“效果链 + 专用处理器”共存。

## 8. 命中判定与空间系统

### 8.1 命中判定的核心原则

正式技能系统的命中判定必须在服务端完成。客户端可以显示技能范围、预测命中、展示轨迹，但真正的“命中了谁”必须由服务端根据权威位置、权威朝向、权威碰撞体和权威地形做裁决。

### 8.2 空间判定对象

服务端应统一维护以下空间判定对象：

- 实体碰撞体
- 技能命中体
- 持续区域体
- 投射物碰撞体
- 地形阻挡体
- 可拦截体

### 8.3 常见命中形态

#### 8.3.1 单体目标命中

适用于锁定技能、普攻、部分装备主动。系统需要校验：

- 目标是否合法
- 是否在有效射程
- 施法过程中是否丢失前置条件

#### 8.3.2 线性命中

适用于刺击、冲击波、箭矢、旋风等技能。关键参数包括：

- 起点
- 方向
- 长度
- 宽度
- 是否穿透
- 首命中是否停止
- 是否被风墙拦截

#### 8.3.3 圆形范围命中

适用于地面落点类技能、爆炸、治疗区、陷阱触发区。关键参数包括：

- 中心点
- 半径
- 延迟时间
- 生效持续时长

#### 8.3.4 扇形命中

适用于近战挥击、喷火、锥形控制技。关键参数包括：

- 起点
- 朝向
- 半径
- 扇形角度

#### 8.3.5 投射物命中

适用于飞行子弹、箭矢、风刃、火球等。投射物必须被视为正式实体，具备：

- 生命周期
- 当前位置
- 速度
- 飞行方向
- 碰撞半径
- 最大飞行距离
- 是否穿透
- 是否可拦截

### 8.4 地形阻挡与风墙类拦截

正式系统中，地形与拦截体必须参与命中计算。否则许多技能规则无法正确表达。

需要重点支持：

- 墙体阻挡位移
- 墙体阻挡弹道
- 风墙拦截可拦截投射物
- 地形生成技能对路径与碰撞的影响

### 8.5 多段命中与命中去重

一些技能可以多段命中同一目标，一些技能则要求同一目标在单次施法中只命中一次。因此技能实例应记录命中历史，并支持以下配置：

- 每实例只命中一次
- 每 Tick 可重复命中
- 命中后需间隔 X 秒才能再次命中
- 同目标伤害递减

### 8.6 目标过滤器的重要性

正式系统不应把“能命中谁”写成散落在各技能中的临时判断，而应提供统一目标过滤器。例如：

- 仅敌方英雄
- 敌方英雄与小兵
- 仅自身
- 自身与友军
- 仅最近目标
- 仅带特定状态目标

这对于终极技能限制、召唤师技能合法目标、装备主动效果分发非常重要。

## 9. 伤害、治疗与状态计算体系

### 9.1 伤害类型

正式系统建议至少支持三种基础伤害类型：

- 物理伤害
- 魔法伤害
- 真实伤害

在此基础上可继续挂标签，例如：

- 技能伤害
- 普攻伤害
- 持续伤害
- 反伤
- 斩杀伤害

### 9.2 伤害计算建议流程

推荐的基础流程如下：

1. 读取技能基础伤害
2. 叠加等级成长值
3. 叠加属性加成系数
4. 叠加 Buff / Debuff 修正
5. 应用增伤 / 减伤标签修正
6. 应用护甲、魔抗、穿透等防御结算
7. 应用护盾吸收
8. 得到最终扣血值

### 9.3 治疗与护盾计算

治疗与护盾不应共用伤害通道做负数结算，而应走独立处理链。因为它们通常会受到不同修正：

- 治疗加成
- 重伤减疗
- 护盾强化
- 护盾类型优先级

### 9.4 Buff / Debuff 叠层与覆盖规则

正式系统必须明确状态的生命周期策略。常见规则包括：

- 新状态覆盖旧状态
- 新状态刷新持续时间但不叠层
- 新状态叠层但不刷新持续时间
- 新状态叠层并刷新持续时间
- 相同类型状态取强覆盖弱
- 相同类型状态互斥

### 9.5 控制效果系统

控制效果不应作为随意字符串挂在 Buff 上，而应有统一枚举和统一语义。正式系统建议至少支持：

- 眩晕
- 沉默
- 禁锢
- 缴械
- 击飞
- 击退
- 压制
- 减速
- 恐惧
- 魅惑
- 致盲
- 失明式普攻失效

不同控制应分别定义对以下行为的影响：

- 是否可移动
- 是否可转向
- 是否可释放技能
- 是否可普攻
- 是否可使用召唤师技能
- 是否可使用装备主动

### 9.6 位移合法性计算

位移是正式技能系统中最容易出错的机制之一，必须服务端权威裁决。系统应支持：

- 冲刺
- 闪现
- 朝目标冲锋
- 朝鼠标点位移
- 击退
- 拉回
- 击飞
- 穿越单位
- 是否允许穿墙

位移裁决时需要检查：

- 起点是否合法
- 目标点是否合法
- 路径上是否存在不可穿越阻挡
- 是否被不可阻挡状态免疫
- 是否被压制或禁锢打断

### 9.7 为什么计算体系必须独立成层

如果把所有伤害、治疗、控制、护盾、位移都直接写进技能类中，系统会快速失控。将计算体系独立出来的好处在于：

- 同类规则统一维护
- 新英雄更容易接入
- 平衡性调整集中可控
- 战斗日志和回放更容易还原
- 更适合服务端权威裁决与审计

## 10. 网络协议、同步策略与客户端预测

### 10.1 协议设计原则

正式产品中的技能协议设计必须遵循四条原则。

第一，客户端发送的是**输入意图**，不是最终结果。  
第二，服务端广播的是**权威事件**，不是让客户端自行推理的半成品。  
第三，高频变化优先走增量事件，稳定状态定期走快照兜底。  
第四，所有关键事件必须具备序列号、战斗时间戳、实例 ID 和幂等处理能力。

### 10.2 建议的客户端请求协议

客户端至少需要支持以下几类战斗输入：

- 移动请求
- 停止请求
- 普攻请求
- 技能施法请求
- 二段施法请求
- 主动取消引导请求
- 召唤师技能请求
- 装备主动请求

其中技能施法请求建议统一为 `CastSpellRequest`，包含：

- 请求 ID
- 战斗房间 ID
- 客户端本地时间戳
- 施法者实例 ID
- 技能槽位
- 技能定义 ID 或技能形态 ID
- 目标单位 ID
- 目标点
- 目标方向
- 客户端预测位置
- 额外上下文参数

### 10.3 建议的服务端广播事件

服务端不应只广播“状态快照”，还应广播显式战斗事件。建议至少包括：

- `SpellCastAccepted`
- `SpellCastRejected`
- `SpellCastStarted`
- `SpellInterrupted`
- `SpellStageChanged`
- `ProjectileSpawned`
- `ProjectileDestroyed`
- `AreaCreated`
- `AreaExpired`
- `StatusApplied`
- `StatusRemoved`
- `DamageApplied`
- `HealApplied`
- `ShieldChanged`
- `DisplacementResolved`
- `DeathOccurred`
- `CombatSnapshot`

### 10.4 为什么需要“事件流 + 快照”双轨同步

如果只发快照，客户端很难还原技能时序、投射物出现时机、中断原因和伤害来源；如果只发事件，客户端在丢包、乱序或长期运行后可能出现局部状态漂移。因此正式系统建议：

- 高频关键表现走事件流
- 稳定权威结果走增量状态同步
- 周期性全量快照做纠偏兜底

### 10.5 客户端预测策略

客户端预测只应覆盖“即时反馈强、作弊风险低、可回正”的部分能力。建议预测以下内容：

- 本地施法动画起手
- 范围指示器变化
- 技能按钮进入预冷却状态
- 可视化投射物预生成
- 本地角色前摇朝向调整
- 局部非权威镜头反馈

不建议客户端权威预测以下结果：

- 最终命中目标
- 最终伤害数值
- 最终控制是否生效
- 最终位移终点是否合法
- 风墙拦截结论
- 击杀与助攻归属

### 10.6 回正与表现修复策略

客户端收到服务端权威结果后，需要有明确的修复路径：

- 若服务端拒绝施法，则撤销本地预冷却与表现性投射物
- 若服务端接受施法但目标不同，则平滑修正命中表现
- 若服务端判定中断，则终止引导表现并进入中断收尾动画
- 若服务端位移终点与本地预测不同，则按策略插值追权威位置

### 10.7 顺序一致性与幂等要求

所有战斗事件都建议带上：

- 房间内递增序列号
- 服务端战斗时间戳
- 技能实例 ID
- 事件实例 ID
- 关联实体 ID

客户端消费事件时必须支持：

- 丢弃旧事件
- 忽略重复事件
- 对缺失事件触发快照回正
- 对乱序事件进行最小化容错处理

## 11. 与 `battle-3d-demo` 的衔接方案

### 11.1 现有可直接复用的模块

当前 `battle-3d-demo` 中，以下能力可以直接演进为正式技能系统的客户端部分：

- `InputController` 作为输入采集入口
- `SkillBar` 作为技能 UI 展示容器
- `Champion.tsx` 作为角色表现宿主

### 11.2 不应直接沿用为正式规则系统的模块

以下能力只能作为 demo 原型参考，不应直接视作正式技能系统：

- 仅靠 `AnimationCommandPayload` 驱动技能
- 仅靠客户端本地动作槽位表示技能存在
- 仅靠 `game:snapshot` 承载完整战斗语义
- 仅在前端临时修改角色血量或状态

### 11.3 客户端需要新增的正式模块

为了接入正式技能系统，建议在前端新增以下模块：

- `ClientSpellInputSystem`
- `SpellAimIndicatorSystem`
- `ClientPredictionSystem`
- `SpellPresentationSystem`
- `ProjectilePresentationSystem`
- `BuffPresentationSystem`
- `CombatEventConsumer`
- `HUDSpellStateAdapter`

### 11.4 动画链路的正式定位

当前 `HeroActionSlot` 与 `AnimationClipRequest` 不应再承担“技能定义”职责，而应降级为“表现执行参数”。正式链路应为：

1. 客户端输入系统触发技能请求
2. 服务端接受施法并广播 `SpellCastStarted`
3. 客户端根据权威事件查到对应技能表现配置
4. 再把表现参数转换成动画请求、特效请求、语音请求

也就是说，正式系统里应是“技能事件驱动动画”，而不是“动画驱动技能存在”。

### 11.5 HUD 的正式演进方向

当前 `SkillBar` 是静态占位。正式接入后，`SkillBar` 应至少展示：

- 技能图标
- 技能等级
- 技能冷却剩余
- 当前可施法状态
- 资源不足提示
- 技能充能层数
- 二段释放窗口提示
- 引导中状态提示
- 强化形态提示

### 11.6 输入系统的正式演进方向

当前 `InputController` 主要处理移动与镜头。正式接入技能系统后，它应被拆为两层：

- 底层输入采集层
- 技能输入语义层

技能输入语义层负责处理：

- 技能按键按下
- 技能按键抬起
- 智能施法与非智能施法
- 鼠标选点
- 鼠标选目标
- 取消施法
- 引导中断
- 二段确认

## 12. 与 `fish-island` 正式项目集成方案

### 12.1 集成定位

`fish-island` 当前已有 Spring Boot 主业务与 Netty WebSocket 基础设施。正式技能系统建议作为独立战斗模块接入，而不是散落在现有聊天、轻互动或非实时玩法逻辑中。

### 12.2 推荐的服务划分

建议至少划分为以下几个职责面：

- 主站业务服务
- 房间与匹配服务
- 实时战斗网关服务
- 战斗模拟与技能执行服务
- 战斗日志与回放服务
- 配置与版本管理服务

### 12.3 Spring Boot 与 Netty 的职责建议

#### 12.3.1 Spring Boot 适合承载

- 账号体系
- 房间创建与入房流程
- 英雄配置下发
- 符文、装备、召唤师技能配置装配
- 战斗结算入库
- 战绩、任务、排行榜等外围业务

#### 12.3.2 Netty / WebSocket 适合承载

- 战斗内高频输入接入
- 权威战斗事件广播
- 断线重连状态恢复
- 低延迟实时消息传输

#### 12.3.3 专门战斗模拟层适合承载

- 战斗 Tick
- 技能生命周期推进
- Buff 系统推进
- 投射物与区域体推进
- 命中与伤害计算
- 死亡与复活逻辑

### 12.4 Redis 的建议使用边界

Redis 可以用来承载：

- 房间索引
- 战斗实例路由
- 短期战斗状态缓存
- 断线重连恢复数据
- 限流与防重处理

但不建议把高频逐 Tick 的完整战斗状态全量依赖 Redis 作为主运行态，而应由战斗实例内存态主导，Redis 辅助兜底与协调。

### 12.5 持久化与回放建议

正式产品中，至少要记录以下内容：

- 对局基础元信息
- 玩家阵容与出装快照
- 技能释放日志
- 伤害与治疗日志
- Buff 变化日志
- 死亡与击杀日志
- 关键帧快照或可回放事件流

这不仅用于复盘与举报审计，也有助于平衡性分析与线上问题定位。

### 12.6 反作弊校验重点

正式技能系统上线后，最容易被客户端伪造的点包括：

- 非法提前施法
- 非法无冷却施法
- 非法资源不足施法
- 非法超距离施法
- 非法目标选择
- 非法位移穿墙
- 非法命中上报

因此这些点都必须在服务端重复校验，客户端任何“结果型上报”都不能直接信任。

## 13. 亚索完整接入示例

亚索是非常适合作为系统验证样板的英雄，因为他同时包含：

- 被动层数与护盾
- 普攻与暴击联动
- 可成长技能形态
- 线性技能
- 投射物技能
- 反制型地形技能
- 目标位移技能
- 目标限制型终极技能
- 击飞联动
- 技能间强耦合关系

如果一套技能系统能比较优雅地承载亚索，那么它通常已经具备较高的正式产品可用度。

### 13.1 亚索的系统建模总览

在正式系统中，亚索不应被实现为“一个写满特殊判断的类”，而应被拆解为：

- 英雄基础属性配置
- 被动定义
- 普攻定义
- `Q/W/E/R` 四个技能定义
- 若干专用状态定义
- 若干专用处理器
- 对应表现配置

建议至少引入以下亚索专属状态或标签：

- `yasuo_flow`
- `yasuo_flow_shield_ready`
- `yasuo_q_stack`
- `yasuo_e_target_mark`
- `yasuo_last_breath_window`
- `yasuo_ultimate_buff`
- `projectile_blockable`

### 13.2 被动：浪客之道

#### 13.2.1 规则拆解

亚索被动建议拆成两个子模块：

- 移动累积流值并形成护盾
- 暴击相关被动修正

#### 13.2.2 工程映射建议

服务端维护 `FlowValue`，通过移动、位移与特定行为累积。达到阈值后，为角色挂上 `flow_shield_ready` 状态。下一次受到满足条件的伤害时：

- 优先消耗护盾
- 触发护盾受击表现
- 移除 `flow_shield_ready`
- 重置流值累计逻辑

暴击相关修正则建议作为被动属性修饰器挂在角色实时属性计算链中，而不是硬编码在普攻逻辑里。

### 13.3 普攻

亚索普攻仍应走统一 `basicAttack` 技能链路，但支持被动、暴击与技能后摇衔接修正。系统需要支持：

- 普攻前摇
- 抬手点命中
- 暴击修正
- 普攻被技能取消或重置
- 普攻后接 `Q` 的流畅衔接

### 13.4 Q：斩钢闪

#### 13.4.1 技能类型

`Q` 建议建模为“方向型瞬发刺击技能”，其释放本体不是投射物，而是近身线性命中体。它还应具备：

- 可叠层
- 受攻速影响的前摇或冷却修正
- 可暴击标签或类普攻标签支持
- 与 `E` 联动生成旋转斩形态的扩展接口

#### 13.4.2 生命周期建议

`Q` 的典型生命周期如下：

1. 客户端按下 `Q`
2. 服务端校验冷却、施法状态与方向合法性
3. 进入短前摇
4. 在关键帧创建线性命中体
5. 命中目标后结算伤害
6. 若命中成功或成功释放，则累积 `Q` 层数
7. 满足条件时将下次 `Q` 形态提升为 `Q3`

#### 13.4.3 伤害判定建议

`Q` 建议走线性近身命中模型，而不是点选单体模型。其命中参数建议配置化：

- 长度
- 宽度
- 命中目标上限
- 是否对同目标只命中一次
- 是否可被闪避/格挡

#### 13.4.4 `Q3` 形态

当 `Q` 层数满足阈值后，下一个 `Q` 不再是普通刺击，而应切换为带旋风效果的强化形态。工程实现上建议：

- 仍属于同一个技能定义
- 通过技能形态状态机切换到 `Q3`
- 生效时创建线性投射物
- 投射物命中后附带击飞效果

这样系统不会把 `Q` 和 `Q3` 强拆成两个互相脱节的技能。

### 13.5 W：风之障壁

#### 13.5.1 技能本质

`W` 的核心不是伤害，而是“创建一个持续存在、可拦截特定投射物的地形 / 拦截实体”。

#### 13.5.2 工程实现建议

服务端在 `W` 生效时创建一个 `WindWallAreaEntity`，该实体具备：

- 位置
- 朝向
- 长度 / 高度 / 厚度
- 存在时间
- 可拦截投射物标签集合
- 拦截回调逻辑

所有带 `projectile_blockable` 标签的投射物在飞行推进时，若与风墙拦截体相交，则：

- 触发拦截事件
- 销毁投射物
- 广播拦截表现事件

#### 13.5.3 为什么必须走统一拦截系统

如果风墙只写成“亚索特殊判断”，后续类似技能会不断复制逻辑。正确方式是建立通用“可拦截投射物 + 拦截体”框架，风墙只是这种机制的一个实例。

### 13.6 E：踏前斩

#### 13.6.1 技能本质

`E` 本质是“选定目标实体后的短位移技能”，同时附带：

- 对目标造成伤害
- 对目标附加短时不可重复冲刺标记
- 与 `Q` 形成组合释放表现

#### 13.6.2 工程实现建议

`E` 的服务端校验重点包括：

- 目标是否存在
- 目标是否为允许冲刺的单位类型
- 目标是否已有亚索 `E` 不可重复标记
- 冲刺路径是否合法
- 路径是否被不可穿越阻挡切断

成功后：

- 创建位移指令
- 推进角色到目标穿越终点
- 对目标结算伤害
- 给目标添加 `yasuo_e_target_mark`

#### 13.6.3 与 `Q` 联动

系统层面不建议把“EQ”写成额外技能，而是让 `E` 的位移阶段与 `Q` 的施法阶段允许形成组合上下文。也就是说：

- 当亚索在特定位移阶段释放 `Q`
- 技能定义读取到 `dashing` 上下文标签
- 切换 `Q` 的命中形态与表现配置

这样可以优雅支持“位移中 `Q` 变体”。

### 13.7 R：狂风绝息斩

#### 13.7.1 技能前置条件

`R` 必须是一个典型的“目标限制型终极技能”。其合法释放前提应至少包括：

- 目标实体存在
- 目标是敌方英雄或允许的目标类型
- 目标当前处于击飞或可被终极技能接续的状态窗口内
- 目标在允许施法距离内

#### 13.7.2 生命周期建议

`R` 的典型流程如下：

1. 客户端检测到合法击飞目标后高亮 `R`
2. 客户端提交目标施法请求
3. 服务端再次验证击飞窗口是否合法
4. 服务端将亚索瞬移 / 贴近到目标合法位置
5. 进入终极技能动画阶段
6. 对目标与规则允许的附加目标进行结算
7. 挂上终极技能后续增益状态

#### 13.7.3 工程关键点

`R` 的关键不在伤害，而在“目标合法性 + 空间落点 + 状态接续”。因此正式实现中，建议专门为它准备：

- 击飞窗口状态定义
- 终极技能目标选择器
- 终极技能落点解算器
- 终极技能后续增益状态

### 13.8 召唤师技能与装备主动在亚索上的接入

亚索接入这套系统后，召唤师技能与装备主动不需要额外特化，只需要：

- 根据玩家局外配置注入 `D/F`
- 根据对局装备生成可用装备主动技能槽
- 统一进入技能栏状态管理
- 统一走服务端权威施法链

例如闪现应直接复用位移技能框架，点燃应直接复用目标型持续 Debuff 框架，火箭腰带类装备主动则复用位移 + 投射物效果链框架。

### 13.9 亚索示例对系统提出的结论

如果系统能正确承载亚索，那么至少说明它已经支持：

- 被动状态驱动
- 普攻与技能共享标签
- 技能形态成长
- 投射物与拦截机制
- 目标位移与不可重复冲刺标记
- 击飞状态依赖终极技能
- 多技能联动与上下文切换

这说明系统已经达到较高的正式产品可用度。

## 14. 分阶段实施路线图

### 14.1 Phase 1：共享协议与技能定义基线

### 14.2 Phase 2：服务端技能执行骨架

### 14.3 Phase 3：Buff、位移、投射物与地形系统

- 拦截体系统

### 14.4 Phase 4：客户端正式接入

目标是让 `battle-3d-demo` 演进为正式客户端战斗表现层，包括：

- 技能输入系统
- HUD 真状态接入
- 表现事件消费器
- 客户端预测与回正

### 14.5 Phase 5：亚索接入与系统验证

目标是用亚索验证系统上限，检查：

- 复杂技能联动是否成立
- 风墙拦截是否正确
- 击飞接续终极技能是否稳定
- 位移合法性是否正确

### 14.6 Phase 6：扩到第二批英雄

建议选择不同范式英雄做横向验证，例如：

- 法师型远程投射物英雄
- 引导型英雄
- 强控制型英雄
- 召唤型或地形型英雄

### 14.7 Phase 7：性能压测与上线准备

包括：

- 技能事件频率压测
- 房间并发压测
- 断线重连验证
- 回放审计验证
- 线上异常日志校验

## 15. 风险、难点与设计取舍

### 15.1 不能只靠配置的原因

如果系统完全依赖配置，复杂技能最终会演变成难以理解的“配置脚本地狱”。对于风墙、位移碰撞、击飞接续、技能上下文变体，必须允许专用处理器存在。

### 15.2 不能只靠客户端预测的原因

如果把命中、伤害、位移结果交给客户端预测并默认成立，那么正式联机环境下会立即出现作弊风险、同步漂移和状态错乱。因此预测只能服务于手感，不能替代权威裁决。

### 15.3 不能把技能表现直接当规则的原因

动画只是表现，不等于命中。特效只是反馈，不等于结算。正式系统必须坚持“规则决定表现”，否则复杂技能一多，系统就会陷入“播了动画但没命中”“明明命中了却没特效”的双向错乱。

### 15.4 最关键的长期收益

这套方案最重要的价值不只是“能做亚索”，而是：

- 新英雄接入成本持续下降
- 复杂技能可以系统化承载
- 前后端边界稳定
- 线上问题容易审计与复盘
- 产品可以长期做平衡调整而不轻易破坏底层结构

---

## 16. 验收标准

一套正式可用的 3D 对战技能系统，至少应满足以下验收标准。

### 16.1 架构层验收

- 前后端职责边界清晰
- 客户端不承担权威裁决
- 技能、状态、投射物、位移、地形拥有统一抽象
- 普攻、被动、召唤师技能、装备主动被纳入同一体系

### 16.2 工程层验收

- 新英雄接入无需修改核心框架
- 新技能接入以配置和处理器扩展为主
- 关键战斗事件可回放、可审计
- 网络协议支持乱序、重放和断线恢复

### 16.3 玩法层验收

- 复杂技能可以被正确表达
- Buff / Debuff / 控制 / 位移 / 投射物 / 地形交互正确工作
- 引导、中断、二段释放、技能联动可稳定运行

### 16.4 亚索示例验收

- 被动护盾逻辑可正确累积与触发
- `Q` / `Q3` 形态切换正确
- `W` 能稳定拦截合法投射物
- `E` 不可重复冲刺标记正确
- `R` 仅能对合法击飞目标释放
- 动画表现、命中判定与伤害结算保持一致

---

## 17. 结论

从正式产品视角看，3D 对战技能系统绝不是一组前端按钮、动画片段和数值表的拼接，而是一套围绕“权威战斗事件流”建立的长期架构。它必须同时满足玩法正确性、联机一致性、扩展性、可维护性与上线可审计性。

当前 `battle-3d-demo` 已经具备优秀的前端主战场底座能力，适合作为正式技能系统的客户端表现承载层；而未来真正上线的技能系统，则应在 `fish-island` 体系中补齐服务端权威战斗模块、统一协议层、计算层与日志审计能力。

以亚索为例，本方案已经证明：只要统一抽象设计得足够稳固，一名高复杂度英雄完全可以在不破坏底层架构的前提下被正确接入，并实现技能演绎、命中判定、伤害裁决与联机同步的一致闭环。

这也是这份正式方案设计文档的最终目标。
